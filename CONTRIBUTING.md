# Правила работы (как именовать коммиты, как делать PR)# Правила разработки (Workflow)

## 1. Работа с ветками
- Ветка `main` — только для стабильного кода. Пушить напрямую в `main` ЗАПРЕЩЕНО.
- Для каждой задачи создаем ветку: `feature/имя-задачи`, `fix/имя-бага` или `develop`.

## 2. Процесс разработки (Git Flow)
1. Разработчик берет задачу из колонки **Todo**, переносит в **In Progress**.
2. Создает ветку, пишет код + **Unit-тесты** (белый ящик).
3. Создает **Pull Request** (PR) из своей ветки в `main`.
4. В PR назначает Тестировщика (Reviewer).

## 3. Тестирование и Review
1. Тестировщик переносит задачу в колонку **Review/Testing**.
2. Проводит **Функциональное тестирование** (черный ящик) по ТЗ.
3. Если всё ок — пишет `Approve` и вливает PR в `main`.
4. Если есть ошибки — возвращает на доработку.

## 4. Стиль кода
1. Используем стандарт: [Например, Google Style Guide или PEP8].
2. Комментарии к методам обязательны (формат не регулируется).

3. Логическое разбиение (Секции)
Длинные файлы должны быть разбиты на логические блоки с помощью визуальных комментариев-заголовков. Это помогает быстро ориентироваться в структуре файла.

    3.1 **Пример для CSS:**
    ``` css
    /* NAVIGATION */

    .nav-item { ... }

    /* FOOTER */

    .footer { ... }
    ```

    3.2 Пример для js:

    ``` JavaScript
    // ОБРАБОТЧИКИ СОБЫТИЙ (EVENT HANDLERS) 

    function handleClick() { ... }
    ```

4. TODO подписывать собственным ником или менем:
    ```js
    //TODO(nickname) ...
    ```

## 5. равила именования коммитов
При комментировании используем стандарт **Conventional Commits**. Каждый коммит должен начинаться с типа и краткого описания в формате: `тип(область): описание`.

| Тип | Когда использовать |
| :--- | :--- |
| **feat** | **Feature** — новая функция (например: добавил возможность выбора проекта). |
| **fix** | **Fix** — исправление ошибки (например: починил SqliteError, поправил CSS). |
| **docs** | **Docs** — изменение документации (README, комментарии в коде). |
| **style** | **Style** — форматирование, отступы (не меняет логику работы, просто чтобы красиво). |
| **refactor** | **Refactor** — изменение кода без добавления функций или исправления багов. |
| **perf** | **Performance** — улучшение производительности (например: ускорил SQL-запрос). |
| **test** | **Test** — добавление или изменение тестов. |

---
**Примеры:**
* `feat(db): add sprint tables and foreign keys`
* `fix(tasks): resolve duplicate task issue in kanban`
* `style(ui): fix navigation bar layout`
